# 小红书封面

小红书 3:4 封面专项。归属 `xiaohongshu`，不是独立同层 skill。

## 推荐先做封面

- 封面决定点击，反馈最快。
- 开源工具需要一句话定位，封面会逼出核心卖点。
- 封面可复用到图文第 1 页、仓库社媒预览、抖音封面。

## 输入

- 工具名。
- 一句话能力。
- 目标用户。
- 最大痛点。
- 1-3 个可证明亮点。
- 截图、logo、demo 图路径。

## 封面类型

先分型，再选构图。不要所有封面都做成大字海报。

| 类型 | 适用内容 | 第一视觉 | 核心判断 |
| --- | --- | --- | --- |
| 产品封面 | App、网站、CLI、插件、SaaS、开源工具 | 品牌名 + 真实截图 | 先证明产品存在且能用 |
| 方法封面 | 教程、流程、经验、Skill 方法 | 方法名 + 步骤感 | 让人觉得可复用、值得收藏 |
| 观点封面 | 判断、反常识、痛点、趋势 | 一句冲突观点 | 让人想验证你说得对不对 |
| 资料封面 | 合集、清单、模板、案例库 | 范围 + 可拿走感 | 让人知道保存后有用 |

Kaji 属于产品封面。它的核心不是“额度跳出来”，而是把 AI working 状态、成本、压力、模型使用收束到一个低打扰的状态栏里。

## 封面公式

- 痛点型：`还在手搓 X？`
- 结果型：`我把 X 变成了 Y`
- 对比型：`以前 X / 现在 Y`
- 清单型：`3 个步骤做出 X`
- 反常识型：`别先写正文，先做封面`

## 产品封面

适用于有真实界面、截图、demo 图的产品。产品封面不靠装饰证明价值，靠截图证明“这个东西存在且能用”。

适用条件：

- App、网站、CLI、插件、工具、SaaS、开源项目有实际界面。
- 用户需要先相信产品形态，而不是先看方法论。
- 有截图、录屏帧、demo 图或产品页主视觉。

结构：

```text
顶部：品牌名 + 一句话定位
中部：一句副标题或轻量批注
主体：真实产品截图，占画面 35%-55%
底部：1-3 个短卖点，每个 6-12 字
```

原则：

- 品牌名要足够可见。产品发布时，品牌是第一信息之一。
- 截图必须是真的产品画面，不用伪 UI。
- 截图要大，不做缩略图；核心界面必须完整，允许轻微裁掉边缘装饰。
- 截图和底部卖点之间保留稳定间距，不能贴边或压字。
- 文字少但重心强，品牌名和定位比截图更先进入视线。
- 卖点只选 1 个主承诺，底部卖点用于补证据。
- 背景干净，白底/浅灰优先；颜色服务品牌，不照抄参考图。
- 手写箭头/批注可以用，但只服务解释，不当装饰。
- 不把截图裁到看不出产品；不让截图压过主标题。
- 不做三等分功能卡；产品截图本身就是中心证据。
- 不学习参考图的红字，而学习它的极简、视觉重心、截图证据和留白。

适合 Kaji 这类产品：

- 品牌：`Kaji` 要放大。
- 主承诺：`AI Coding 一站式状态栏` 这类产品定位，优先于单个痛点。
- 截图：菜单栏 popover、用量面板或真实使用界面。
- 底部卖点：额度可见、状态收束、少打断。

产品封面检查：

- 截图来自真实产品，不是临时画的概念面板。
- 品牌名、定位、截图三者说的是同一个产品。
- 主色只做强调，不照抄参考图配色。
- 画面有一个明确视觉重心，不平均铺满。
- 底部卖点平行、短、可证明。
- 截图和文字之间的间距稳定。

## 方法封面

适用于教程、流程、经验总结、Skill 方法。方法封面不是产品发布图，重点是“我照着做也能复用”。

结构：

```text
顶部：方法名或结果承诺
中部：3-5 个步骤、流程箭头或示例结果
底部：适用人群 / 使用场景 / 可保存提示
```

原则：

- 标题说结果，不说“干货分享”。
- 步骤少而具体，避免把正文提纲整页塞进封面。
- 可以做流程感、卡片感、清单感；不需要强行放产品截图。
- 数字必须真实。没有确定数量时，用“几个”“一套”“从 0 到 1”。
- 视觉要有收藏感，但不要堆贴纸、emoji、荧光笔。

## 观点封面

适用于判断、反常识、争议点、痛点表达。观点封面靠冲突打开注意力，但不能靠夸张造假。

结构：

```text
顶部：对象或场景
中部：一句明确判断
底部：补一行解释或反差
```

原则：

- 观点必须可辩论，不能只是空泛形容。
- 主标题可以更大，但仍要控制在 3-10 字。
- 适合做对比、划线、轻量批注；不要把视觉做成情绪海报。
- 不写“全网都错了”“没人告诉你”这类廉价句式。

## 资料封面

适用于模板、清单、案例库、参考合集。资料封面要让人知道保存后能直接用。

结构：

```text
顶部：资料范围
中部：模板/案例/清单的局部预览
底部：适用场景 + 获取方式
```

原则：

- 范围要具体：平台、对象、场景至少说清一个。
- 有真实数量才写数量；没有就写“案例库”“清单”“模板”。
- 预览比装饰重要，展示 2-4 个真实片段即可。
- 不把资料封面做成产品截图，除非资料本身就是产品界面。

## 开源工具默认结构

```text
主标题：AI 社媒 Skill 开源了
副标题：文案 / 封面 / 图文卡片 一套跑通
角标：开源工具
底部：social-media-skills
```

## 视觉方向

- 比例：3:4。
- 风格：清爽工具感 + 小红书强标题；不要纯代码仓库截图。
- 主体：真实产品形态、截图抽象、状态面板、卡片骨架。
- 色彩：白底/浅灰底 + 一个强调色 + 黑色大字。
- 字体：主标题粗黑，副标题中等，平台名用小标签。
- 构图：不要死居中；优先“左/上大字 + 右/下产品形态”。
- 信息密度：主标题、副标题、角标三层以内；底部只放工具名或仓库名。

## 视觉检查

从 `design-taste-frontend` 迁移到封面的检查：

- 不用 AI 紫蓝渐变当默认主视觉。
- 不用三等分功能卡当主画面。
- 不用装饰点、状态点、版本号堆满画面。
- 不写假精确数字、假榜单、假用户反馈。
- 不用纯黑纯白；用炭黑、浅灰、一个品牌强调色。
- 主标题不超过 10 字；副标题不超过 20 字。
- 中文优先黑体体系；不要为了“高级感”硬上衬线。
- 视觉主体必须像产品，不像装饰插画。
- 产品封面必须有真实截图；没有截图就停止补素材。

## 封面评分

每张封面 10 分制：

- 2 分：主标题 1 秒可读。
- 2 分：承诺具体，不是泛泛“效率提升”。
- 2 分：产品形态可见，截图/状态/菜单栏有真实感。
- 2 分：层级干净，三层信息以内。
- 2 分：无 AI 味，避开渐变、emoji、假数据、空泛标签。

低于 8 分，不进入 case。


## 发布文案骨架

```markdown
我做了一个 [工具类型]：`[工具名]`。

它解决的问题不是“更多功能”，而是：[真实痛点]。

适合：
1. [人群]
2. [场景]
3. [已有行为]

现在能做：
1. [能力一]
2. [能力二]
3. [能力三]

我最想保留的是：[产品原则]。

GitHub：`[repo]`
```
