# 产品封面

适用于有真实界面、截图、demo 图的产品。产品封面不靠装饰证明价值，靠截图证明“这个东西存在且能用”。

## 适用条件

- App、网站、CLI、插件、工具、SaaS、开源项目有实际界面。
- 用户需要先相信产品形态，而不是先看方法论。
- 有截图、录屏帧、demo 图或产品页主视觉。

没有截图时，先向用户要。不要用伪 UI 硬撑。

截图证据优先级：

```text
真实产品截图 > 真实生成结果 > demo 输出页 > CLI 输出 > README 首屏 > GitHub 文件列表
```

对 skill 仓库、prompt 库、agent 规则库这类无传统 UI 的开源产品，优先展示真实生成结果或 case 成品。GitHub 文件列表只能做辅助证据，不能当主视觉。

## 结构

```text
顶部：品牌名 + 一句话定位
中部：一句副标题或轻量批注
主体：真实产品截图，占画面 35%-55%
底部：1-3 个短卖点，每个 6-12 字
```

## 原则

- 品牌名要足够可见。产品发布时，品牌是第一信息之一。
- 主承诺说产品定位，不只说局部功能。
- 截图必须是真的产品画面，不用伪 UI。
- 截图要大，不做缩略图；核心界面必须完整，允许轻微裁边缘装饰。
- 截图不得裁掉核心界面；优先 `contain` 或人工确认核心区完整，不用 `slice` 直接吞掉关键信息。
- 截图和底部卖点之间保留稳定间距，不能贴边或压字。
- 文字少但重心强，品牌名和定位比截图更先进入视线。
- 卖点只选 1 个主承诺，底部卖点用于补证据。
- 背景干净，白底/浅灰优先；颜色服务品牌，不照抄参考图。
- 手写箭头/批注可以用，但只服务解释，不当装饰。
- 不做三等分功能卡；产品截图本身就是中心证据。
- 不学习参考图的大红字，学习极简、视觉重心、截图证据和留白。

## Kaji 锚点

- 品牌：`Kaji` 要大。
- 定位：`AI Coding 一站式状态栏` 这类产品定位，优先于单个痛点。
- 截图：菜单栏 popover、用量面板或真实使用界面。
- 底部卖点：额度可见、状态收束、少打断。

## 开源工具产品

开源工具也按产品处理，只要有真实界面、生成结果、CLI 输出、demo 页或截图。

```text
顶部：工具名 + 一句话定位
中部：真实截图 / 生成结果 / demo 画面
底部：开源、可安装、可改造中选 1-3 个证据
```

不要把 GitHub 文件列表当主视觉，除非仓库本身就是要展示的产品界面。

## 检查

- 截图来自真实产品，不是临时画的概念面板。
- 截图来源可解释：来自用户提供、产品实拍、已有 case 成品、demo 输出或 CLI 输出。
- 品牌名、定位、截图三者说的是同一个产品。
- 主色只做强调，不照抄参考图配色。
- 画面有一个明确视觉重心，不平均铺满。
- 底部卖点平行、短、可证明。
- 截图和文字之间的间距稳定。
